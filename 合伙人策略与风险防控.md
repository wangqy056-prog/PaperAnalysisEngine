## 四、怎么验证合伙人

> **先共事，再合伙。不做"闪婚"。**

```
Step 1：小项目试水（1-2 周，无股权承诺）
  → 让对方独立完成一个小功能（如 Docker 化部署）
  → 观察：交付质量、沟通效率、主动性

Step 2：兼职合作（1-3 个月，顾问费或无薪但有期权目标）
  → 设定明确的里程碑（如"上线公开版"）
  → 观察：承诺兑现率、面对困难的态度

Step 3：正式合伙
  → 谈股权比例、vesting、退出条款
  → 签合伙人协议
```

## 六、降低合伙风险的三个原则

**原则 1：能一个月验证的事，不要签一年合同**

先让对方用业余时间做一个可交付的小功能。行就继续，不行就礼貌结束。一个月能看出来的事：代码质量、沟通习惯、是不是"鸽子精"。

**原则 2：不谈钱的时候最危险**

不谈薪酬/股权的"兄弟合伙"迟早崩。哪怕现在是兼职没工资，也要明确记录谁贡献了什么，未来怎么分。推荐用 `slicingpie.com` 的动态股权模型，谁贡献谁积累股权份额。

**原则 3：关键资源自己控制**

- 域名 → 你自己注册
- GitHub Org → 你自己是 Owner
- 数据库 → 你自己有备份
- API Key → 你自己持有主 key

不是不信任对方，而是**确保你在任何情况下都能继续跑这个项目**。

**这会大幅提高对方认真对待的概率。**

## 八、防抄袭与防叛变策略

### 8.1 残酷真相

软件项目**没有百分百的防护**。代码可以抄，API 公开，算法论文里有——不依赖任何人的主观意愿。

**但换个角度：如果一个人看了你的东西就能独立重做一遍，说明护城河还不够深。**

### 8.2 五层防护体系

```
第 1 层：法律防火墙（最基础，威慑力中等）
第 2 层：信息防火墙（日常防护）
第 3 层：技术防火墙（让对方"抄不了"）
第 4 层：生态防火墙（让对方"抄了也没用"）
第 5 层：不可替代性防火墙（让对方"不想抄"）
```

---

#### 第 1 层：法律防火墙

| 措施 | 何时用 | 效力 |
|------|--------|------|
| **NDA（保密协议）** | 第一次深度交谈前 | 威慑 > 执行（个人对个人打官司太贵，且国内法院执行周期长） |
| **合伙人协议（含竞业）** | 确定合作时签 | 中等效力——但国内竞业需有补偿（通常月薪30%）否则可能无效 |
| **IP 归属协议** | 合作开始时签 | 明确所有产出归公司/项目，不是个人 |
| **开源许可证** | 代码公开时 | AGPL → 别人必须开源他们的修改版 |

> ⚠️ **诚实说**：如果前 3 层（法律+信息+技术）都没拦住对方，打官司也不是选项——早期团队没时间也没钱打官司。**真正的防护是让对方觉得没必要抄，而不是抄了以后告他。**

---

#### 第 2 层：信息防火墙

**不是什么都说。**

| 可以公开 | 不要分享 |
|---------|---------|
| 产品愿景、功能列表 | 评级算法中具体参数值（λ=0.693/5这类） |
| 技术架构图 | 种子用户的反馈和痛点 |
| 竞品分析 | 未发布功能的详细设计文档 |
| GitHub 开源代码 | 专利-论文映射的具体数据源和匹配逻辑 |
| | 潜在企业客户的联系方式和报价 |
| | 增长策略的具体执行细节和时间表 |

**实操**：跟潜在合伙人聊的时候，可以展示产品、分享商业策略文档，但**评级引擎的校准参数、专利映射的数据管线逻辑**这两样先稳住不说。

---

#### 第 3 层：技术防火墙（核心）

**让产品依赖你独有的东西，别人抄不走。**

```
┌─────────────────────────────────────────────┐
│ 你独有的                                     │
├─────────────────────────────────────────────┤
│ ✅ 571篇论文 + 40K引用关系数据集              │
│    → 别人要重爬需要几天                      │
│                                              │
│ ✅ 评级参数校准经验                           │
│    → 参数可以抄，但"调参的秘密"在经验中       │
│                                              │
│ ✅ 专利-论文映射专属数据集                     │
│    → 构建需要7-11天，但第一手数据只有你有       │
│                                              │
│ ✅ 用户行为数据（未来）                        │
│    → 每次搜索、点击、收藏都在训练推荐引擎      │
│    → 后进者即使抄了代码，也是冷启动             │
└─────────────────────────────────────────────┘
```

**最关键的动作**：在任何人参与之前，尽快把**专利-论文映射**做出来。虽然 BigQuery 数据公开可用，但真正的护城河在于：① 你先做出来的消歧阈值和匹配策略（别人即使有同样数据源，不知道参数）；② 用户验证后持续校准的质量提升（先发优势的数据飞轮）。

---

#### 第 4 层：生态防火墙

**让对方抄了功能也没用户。**

| 防守手段 | 原理 |
|---------|------|
| **Zotero 插件率先发布** | 用户装了就离不开，迁移成本高 |
| **开放 API 吸引第三方** | 别的工具接入了你的 API → 换平台成本高 |
| **学术圈口碑** | "大家都用 PAE" → 网络效应，新进入者没用户 |
| **用户贡献的数据** | 标签、纠错 → 社区积累的数据是新进入者没有的 |

---

#### 第 5 层：不可替代性防火墙

**最强的防守：让对方觉得"跟着你干比单干划算"。**

```
为什么选你而不去单干？
─────────────────────────
✅ 你不是"产品经理提需求"——你会搭框架、写代码、有571篇数据的成品
✅ 你不是"我有一个想法"——你有完整的竞品分析、商业模型、技术路线
✅ 你不只是"创始人"——你是这个项目最懂论文评级的人
✅ 跟着你干 = 快速上线、快速验证、快速赚钱
   自己单干 = 重做一遍你已做好的事情 = 浪费6个月
```

**持续让对方觉得"离了你不行"的方法**：
- 掌握核心数据（论文库、专利映射）
- 掌握核心关系（种子用户、学术顾问）
- 掌握核心知识（评级算法调优经验）
- 持续快速迭代（让对方追不上）

## 十、单兵筑墙计划：一个人 + AI Agent 能做多少

### 10.1 目标

> 在不依赖任何合伙人的前提下，用 AI Agent 辅助，把项目推到"抄起来不划算"的程度。

### 10.2 为什么单兵 + AI 可行

| 传统瓶颈 | AI 能做什么 | 你的角色 |
|---------|-----------|---------|
| 写代码慢 | Agent 生成 80% 代码 | 审查 + 架构决策 + 调参 |
| 不会 Docker | Agent 写 Dockerfile | 测试部署 |
| 写文案/文档慢 | Agent 起草 + 润色 | 最终定稿 |
| 数据清洗繁琐 | Agent 写 ETL 脚本 | 验证数据质量 |
| 前端不够好看 | Agent 生成组件代码 | 调整交互细节 |

> **你的核心优势不是"写得快"，而是"知道该做什么"——这是 AI 替代不了的。**

### 10.3 分阶段路线图

```
[Week 1-2]  [Week 3-4]    [Week 5-6]    [Week 7-8]
 工程化      数据壁垒       分发壁垒       品牌壁垒
   ↓            ↓              ↓             ↓
 Docker     专利-论文       Zotero        落地页
 .env       映射完成        插件发布       开源协议
```

### Phase 1：工程化（1-2 周）— "看起来像正经产品" 🔴 必须先做

**为什么先做这个**：别人打开你的 GitHub，第一眼看到的是代码质量、部署文档、API 文档。没有这些，再好的算法也是"玩具"。

> ⚠️ **Phase 1 也是商业化准入门槛**：T1（API Key 硬编码）不修 → 代码泄露 = 密钥泄露；T3（无 Docker）不修 → Enterprise 客户无法自助部署 → 直接拒绝合作；T4（零测试）不修 → 迭代出 Bug 摧毁用户信任。

| 任务 | AI 能帮什么 | 产出 | 耗时 |
|------|-----------|------|------|
| T1 修复：API Key → .env | 生成配置模板 | `.env.example` + `config.py` 改版 | 0.5天 |
| T3 修复：Docker 化 | **Agent 写完整 Dockerfile + docker-compose** | 一条命令启动全栈 | 1 天 |
| 清理 engine/ 旧代码 | Agent 对比两套代码，标注差异 | 统一到 backend/ | 0.5天 |
| API 文档完善 | Agent 补 Swagger 注释 | `localhost:8000/docs` 完整 | 0.5天 |
| CI/CD | Agent 写 GitHub Actions | 自动测试 + 自动部署 | 0.5天 |
| 一键安装脚本 | Agent 写 install.sh | 新用户 3 步搞定 | 0.5天 |

**Phase 1 完成后**：任何人 clone 代码 → `docker-compose up` → 浏览器打开就能用。

```
           ├── 落地页 + GitHub README
```

> ⚠️ **为什么从 6 周改成 8-10 周**：AI 能加速 80% 的工作（写代码、写文档、写配置），但剩下 20%——架构决策、参数调优、真机测试、内容发布后的社区互动——必须你亲自做。给每个阶段留缓冲，避免"连续加班 6 周然后 burnout"。

### 10.5 各阶段的"抄袭成本"

| 时间点 | 项目状态 | 抄袭者需要做什么 | 需要多久 |

| **Week 2** | + Docker + 5000 论文 | 重写代码 + 重爬数据 | 3-4 周 |

> **达到第 11 周状态后，单兵抄袭的 ROI 极低——与其重做 10,000 行代码 + 爬数据 + 建映射 + 写插件 + 追用户，不如直接加入你。**
