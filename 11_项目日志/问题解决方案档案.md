## #1 — ImportError: cannot import name 'get_paper_by_id'

**日期：** 2026-06-28
**严重程度：** 高（阻断Web UI启动）

### 问题描述
Streamlit Web UI 启动时立即报错，无法加载任何页面。

### 错误信息
```
ImportError: cannot import name 'get_paper_by_id' from 'db'
```

### 根因分析
`app.py` 中导入的函数名与 `db.py` 实际定义不匹配：
- `app.py` 导入了 `get_paper_by_id`、`get_paper_rating`、`get_paper_tags`、`insert_tags`
- `db.py` 实际定义的是 `get_paper`，且没有 `get_paper_rating`、`get_paper_tags`、`insert_tags` 函数

这是因为在开发 Web UI 时，函数名是按预期设计的，但 `db.py` 实际实现时用了不同的命名。

### 解决方案
1. 将 `app.py` 中的 `get_paper_by_id` 改为 `get_paper`
2. 移除不存在的 `get_paper_rating`、`get_paper_tags`、`insert_tags` 导入
3. 在 `app.py` 中内联辅助函数，直接查询数据库：
   - `_get_rating_for_paper(paper_id)` — 查询 ratings 表
   - `_get_tags_for_paper(paper_id)` — 查询 tags + paper_tags 联表
   - `_save_tags(paper_id, tags)` — 插入标签和关联

### 修改文件
- `backend/app.py` — 修改导入语句，新增内联辅助函数

### 预防措施
开发前先确认 db.py 中已定义的函数清单，保持导入与定义一致。

## #4 — OpenAlex SSL 连接错误

**日期：** 2026-06-28
**严重程度：** 中（偶发，影响稳定性）

### 问题描述
调用 OpenAlex API 时偶发 SSL 连接错误。

### 错误信息
```
SSLError(SSLEOFError(8, 'EOF occurred in violation of protocol (_ssl.c:1007)'))
```

### 根因分析
网络环境（代理/防火墙）导致 SSL 握手被中断。

### 解决方案
在 `_request_with_retry` 中添加 SSL 降级机制：
1. 第一次尝试正常 SSL（verify=True）
2. SSL 失败后降级为不验证证书（verify=False）
3. 最多重试 3 次，每次间隔 2 秒
4. 禁用 InsecureRequestWarning 警告

### 修改文件
- `backend/paper_fetcher.py` — `_request_with_retry` 函数
- `backend/fetch_references.py` — 同样的 SSL 降级逻辑

### 代码片段
```python
except requests.exceptions.SSLError:
    kwargs["verify"] = False
    resp = session.get(url, timeout=timeout, **kwargs)
```

## #10 — UnicodeEncodeError: GBK 编码问题

**日期：** 2026-06-28
**严重程度：** 高（导致批量导入崩溃）

### 问题描述
批量导入论文时进程崩溃，退出码1。

### 错误信息
```
UnicodeEncodeError: 'gbk' codec can't encode character '\u2009' in position 36: illegal multibyte sequence
```

### 根因分析
Windows 控制台默认使用 GBK 编码，无法处理论文标题中的特殊字符（如 `\u2009` 薄空格、`ë` 等）。`print()` 函数尝试用 GBK 编码输出时抛出异常，由于 `import_topic` 函数内没有 try/except，异常导致整个进程崩溃。

### 解决方案
在脚本入口处添加 UTF-8 编码重配置：

```python
import sys
if sys.platform == "win32":
    try:
        sys.stdout.reconfigure(encoding='utf-8', errors='replace')
        sys.stderr.reconfigure(encoding='utf-8', errors='replace')
    except Exception:
        pass
```

### 修改文件
- `backend/import_papers.py` — 添加编码修复
- `backend/resume_import.py` — 添加编码修复
- `backend/fetch_references.py` — 添加编码修复

### 预防措施
所有 Windows 上运行的 Python 脚本，入口处统一添加 UTF-8 编码重配置。已固化为项目模板。

4. 移除仍然孤立的节点

### 结果
- 优化后：80节点 / 57边 / 平均度数1.43

### 修改文件
- `backend/knowledge_graph.py` — `build_similarity_network` 函数

## #14 — ON CONFLICT clause does not match any PRIMARY KEY or UNIQUE

**日期：** 2026-06-28
**严重程度：** 高（推荐引擎浏览记录功能不可用）

### 问题描述
推荐引擎的 `record_view()` 函数调用时报错，无法记录论文浏览历史。

### 错误信息
```
sqlite3.OperationalError: ON CONFLICT clause does not match any PRIMARY KEY or UNIQUE constraint
```

### 根因分析
`viewing_history` 表在 `db.py` 中定义时缺少 `UNIQUE` 约束，但 `record_view()` 使用了 `ON CONFLICT(paper_id) DO UPDATE` 语法，该语法要求字段必须有 `UNIQUE` 或 `PRIMARY KEY` 约束。

由于 `CREATE TABLE IF NOT EXISTS` 不会修改已存在表的结构，老表缺少约束。

### 解决方案
创建 `migrate_db.py` 迁移脚本：
1. 重命名旧表 `viewing_history` → `viewing_history_old`
2. 用新结构（含 `UNIQUE`）创建新表
3. 迁移数据
4. 删除旧表

同样的修复应用于 `user_ratings` 和 `user_preferences` 表。

### 修改文件
- `backend/db.py` — `viewing_history` 表定义添加 `UNIQUE`
- `backend/migrate_db.py` — 临时迁移脚本（执行后删除）

### 代码片段
```python
# 迁移脚本核心逻辑
cursor.execute("ALTER TABLE viewing_history RENAME TO viewing_history_old")
cursor.execute("""
    CREATE TABLE viewing_history (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        paper_id TEXT NOT NULL UNIQUE,
        ...
    )
""")
cursor.execute("INSERT OR IGNORE INTO viewing_history ... SELECT ... FROM viewing_history_old")
cursor.execute("DROP TABLE viewing_history_old")
```

### 预防措施
使用 `ON CONFLICT ... DO UPDATE` 语法前，确保字段有 `UNIQUE` 或 `PRIMARY KEY` 约束。对于已存在的表，用 ALTER + 重建的方式迁移结构。

## #15 — StreamlitDuplicateElementKey (收藏夹页面)

**日期：** 2026-06-28
**严重程度：** 中（收藏夹页面崩溃）

### 问题描述
收藏夹页面点击查看论文时报错：`StreamlitDuplicateElementKey: There are multiple elements with the same key='fv_arxiv_2606.27372v1'`

### 根因分析
收藏夹列表循环渲染按钮时，key 使用 `f"fv_{p['id']}"`，当论文列表中有重复数据（SQL JOIN 产生重复行）时，同一篇论文的按钮 key 冲突。

### 解决方案
双重保险：
1. **SQL 层**（[favorites_manager.py](file:///d:/Trae/chat/Paper_Analysis_Engine/backend/favorites_manager.py#L193-L204)）— 添加 `SELECT DISTINCT` + `GROUP BY p.id` 去重
2. **UI 层**（[app.py](file:///d:/Trae/chat/Paper_Analysis_Engine/backend/app.py#L1298-L1335)）— 所有按钮 key 添加索引前缀：`fv_{idx}_{id}` / `rm_{idx}_{id}` / `note_{idx}_{id}` / `sn_{idx}_{id}`

### 预防措施
Streamlit 循环渲染组件时，key 应包含索引（如 `{idx}_{id}`）确保唯一性，即使数据源去重也加双重保险。

切换功能导航后，新页面下方仍显示上一个页面的内容（如领研网论文列表残留在其他页面）。

### 根因分析
1. **session_state 残留** — `lr_papers`、`selected_paper` 等页面特有状态在切换导航后未清理，导致下次进入同一页面时显示旧数据，或与其他页面数据串扰
2. **浏览器滚动位置未重置** — 切换到较短页面后，浏览器仍停留在上次的滚动位置，下方显示旧页面残留

### 解决方案
在主逻辑路由处添加页面切换检测（[app.py](file:///d:/Trae/chat/Paper_Analysis_Engine/backend/app.py#L1559-L1604)）：

```python
# 检测页面切换
page_changed = (st.session_state.current_page != page)
if page_changed:
    # 清理所有页面特有的临时状态
    for key in ["selected_paper", "fav_paper", "lr_papers",
                "batch_results", "search_results", "rec_results"]:
        st.session_state.pop(key, None)

# 仅在页面切换时滚动到顶部
if page_changed:
    st.components.v1.html(
        """<script>
            window.parent.document.querySelector('section.main')
                .scrollTo({top: 0, behavior: 'instant'});
        </script>""",
        height=0,
    )
```

### 关键设计
- **仅在 page_changed 时清理** — 避免页面内交互（如点击收藏按钮）也触发清理
- **仅在 page_changed 时滚动** — 页面内 rerun 不重置滚动位置，保持用户浏览位置
- **清理范围** — 所有页面特有的临时状态（selected_paper/fav_paper/lr_papers/batch_results/search_results/rec_results）

### 预防措施
所有页面特有的临时状态应统一在页面切换时清理。新页面如需使用 session_state 缓存，应将 key 加入清理列表。

### 后续修复（2026-06-28）
用户反馈"领研网论文还在其他页面残留"且"不会自动滚到顶部"。原因：
1. 之前的 JS 用 `behavior: 'instant'` 非标准值，浏览器不识别导致滚动失败
2. `st.components.v1.html` 的 iframe 在页面内容后渲染，需要延迟执行

**最终修复**：
- 用标准 `scrollTo(0, 0)` 代替 `scrollTo({top:0, behavior:'instant'})`
- 用 `setTimeout(..., 200)` 延迟 200ms 等页面渲染完成后再滚动
- 用 `try-catch` 防止 JS 异常干扰渲染
- 清理范围从 2 个 key 扩展到 6 个（含 `lr_papers`/`batch_results`/`search_results`/`rec_results`）

### 第三次修复（2026-06-28）
用户反馈"领研网残留还在每个功能下面，点导航不会自动定位到顶端"。深层原因：
1. **`st.components.v1.html` 的 iframe 干扰 DOM** — iframe 创建独立文档，可能阻止 Streamlit 正确清除旧页面 DOM，导致残留
2. **`<img onerror>` bug** — 用了有效的 data URI 作为 src，图片成功加载，`onerror` 永远不触发，JS 根本没执行

**最终方案**：
- 移除 `st.components.v1.html`（彻底消除 iframe 干扰）
- 改用 `st.markdown(unsafe_allow_html=True)` + `<img onload>`（onload 在图片成功加载时触发，确保 JS 执行）
- 重启 Streamlit 服务确保新代码生效

**关键教训**：
- `st.components.v1.html` 创建的 iframe 可能干扰 Streamlit 的 DOM 清除机制
- `<img onerror>` 需要无效的 src 才能触发；有效的 src 要用 `onload`
- 修改代码后必须重启 Streamlit 服务，不能只依赖自动重载

### 最终决策（2026-06-28）
经过 3 次修复尝试仍无法彻底解决页面切换 DOM 残留问题。**决定放弃 Streamlit，迁移到其他技术栈**。

Streamlit 的固有限制：
1. 单文件应用结构，页面切换机制不灵活
2. session_state 管理复杂，容易状态残留
3. DOM 控制能力弱，无法强制清除旧页面内容
4. JS 注入受限（`st.components.v1.html` 的 iframe 干扰渲染）
5. 不适合复杂的多页面交互应用

**Streamlit 定位**：MVP 阶段快速验证功能的过渡方案，不适合作为最终产品。

**替代方案**：见技术决策记录 ADR-010

## #17 — Vue Router 导航卡死（推荐页 authors 字段类型错误）

### 错误信息
```
TypeError: (rec.authors || (intermediate value) || []).slice(...).join is not a function
    at Recommendations.vue:57
    at renderFnWithContext (...)
    at renderSlot (...)
    at <ElCard>

  at <RouterView>
  at <ElMain>

# 后续连锁错误（每个都让路由更难恢复）：

    at unmountComponent (...)
    ...（重复 7 次）
```

### 根因分析
1. **后端**：[recommendation_engine.py](file:///d:/Trae/chat/Paper_Analysis_Engine/backend/recommendation_engine.py) 的 `recommend_by_rating` / `recommend_by_trending` / `recommend_by_recent` / `recommend_by_field` 函数直接 `SELECT p.authors FROM papers ...`，SQLite 返回的 `authors` 字段是 JSON 序列化后的字符串（如 `'["张三", "李四"]'`），但函数没解析回数组就直接返回。
2. **前端**：[Recommendations.vue:57](file:///d:/Trae/chat/Paper_Analysis_Engine/frontend/src/views/Recommendations.vue#L57-L58) 的模板表达式 `(rec.authors || rec.paper?.authors || []).slice(0, 3).join(', ')` 假设 `authors` 是数组。对字符串：
   - `.slice(0, 3)` 返回子串（如 `'["a'`），不是数组
   - 子串调用 `.join(', ')` → `TypeError: join is not a function`
3. **连锁崩溃**：渲染函数抛错向上传播到 `<RouterView>`，触发 `Unhandled error during execution of component update`，导致 Vue 内部 vnode 状态不一致（`Cannot destructure property 'type' of 'vnode' as it is null`），后续任何路由切换都无法渲染新组件。

### 解决方案

**后端修复**（[recommendation_engine.py](file:///d:/Trae/chat/Paper_Analysis_Engine/backend/recommendation_engine.py#L468-L501)）：
新增 `_normalize_paper()` 辅助函数，在统一入口 `get_recommendations` 返回前把 `authors/fields/ref_ids/tags` 字段从字符串解析为数组：

```python
def _normalize_paper(p: Dict) -> Dict:
    for key in ("authors", "fields", "ref_ids", "tags"):
        v = p.get(key)
        if isinstance(v, str):
            try: p[key] = json.loads(v)
            except (json.JSONDecodeError, ValueError): p[key] = []
        elif v is None: p[key] = []
        if not isinstance(p[key], list): p[key] = []
    return p

def get_recommendations(strategy="hybrid", top_n=20, **kwargs):
    # ... 调用具体策略
    return [_normalize_paper(p) for p in recs]  # 统一规范化
```

**后端修复**（[db.py:search_papers](file:///d:/Trae/chat/Paper_Analysis_Engine/backend/db.py#L302-L322)）：
同样问题，`search_papers` 也直接返回数据库 dict 没解析 JSON 字段，添加相同的归一化逻辑（影响 Search/Leaderboard 页面）。

**前端防御**（[Recommendations.vue](file:///d:/Trae/chat/Paper_Analysis_Engine/frontend/src/views/Recommendations.vue#L89-L95)）：
增加 `authorsList()` 辅助函数做双保险：

```javascript
const authorsList = (rec) => {
  const a = rec.authors || rec.paper?.authors || []
  return Array.isArray(a) ? a : []
}
```
模板改用 `authorsList(rec).slice(0, 3).join(', ')`。

### 验证结果
用 Chrome DevTools Protocol 模拟用户操作，端到端验证：
- `/` → `/recommendations` → `/leaderboard` → `/`
- `allSwitched: true`，`__errors: []` ✅

### 关键教训
1. **后端 API 必须保证数据类型一致** — 数据库 JSON 字段在返回前必须解析为原生类型，不能让前端猜测字符串还是数组
2. **前端模板表达式不能假设数据结构** — 用 `Array.isArray()` 防御性检查，特别是涉及 `.slice()/.join()` 这类数组方法
3. **单个页面渲染错误会"感染"整个 RouterView** — Vue 3 的 RouterView 在子组件抛错时可能进入不可恢复状态，导致后续所有路由切换失效；这种"卡死"现象是 RouterView 崩溃的典型特征
4. **诊断思路** — 用 CDP（Chrome DevTools Protocol）远程执行 JS 收集 `Runtime.exceptionThrown` 事件，能精确还原用户操作时的错误链

### 预防措施
- 数据库读取层（`db.py`）和业务层（`recommendation_engine.py` 等）的 JSON 字段统一在边界处解析
- 所有 Vue 模板中的数组方法调用前用 `Array.isArray()` 或独立辅助函数包裹
- 复杂页面建议在 `<script setup>` 顶部加 `onErrorCaptured` 兜底，避免错误冒泡到 RouterView

edges.append({

## #22 — AI 分析使用逻辑错误（切换类型重复调用 LLM 浪费时间）

**日期：** 2026-06-28
**严重程度：** 高（每次切换都重新调 LLM，6 次分析要等 6 倍时间）

### 问题描述
用户反馈："点击下面 AI 分析的时候只要跳转到分析页面就可以了不需要用 AI 重新再生成一遍摘要等半天才切换过来而是在用户手动点开始分析之后把摘要翻译解读这些全部生成好而不是每切换一个类型就重新分析一遍浪费巨多时间"

### 根因分析
[AIAnalysis.vue](file:///d:/Trae/chat/Paper_Analysis_Engine/frontend/src/views/AIAnalysis.vue) 的旧逻辑：
1. `onSearchPaper` 查询论文后自动调用 `onAnalyze` 生成默认类型（摘要）→ 用户从详情页跳过来就被迫等一次 LLM 分析
2. `el-radio-group @change="onTypeChange"` 在切换类型时调用 `onAnalyze` → 每切一次就调一次 LLM API
3. 结果只存当前类型的 `result.value` 字符串，没有缓存 → 切换后旧结果丢失，必须重新生成

**核心问题**：结果没有缓存，每次切换都重新调用 LLM。

### 正确逻辑（用户期望）
1. 从论文详情跳转 → 只加载论文信息，不自动分析
2. 用户点"开始分析" → 一次性并行生成所有 6 种分析，缓存结果
3. 切换类型按钮 → 只切换显示已缓存的结果，不再调 API

### 解决方案 — 重写 [AIAnalysis.vue](file:///d:/Trae/chat/Paper_Analysis_Engine/frontend/src/views/AIAnalysis.vue)

**1. 结果缓存机制**
```javascript
// 结果缓存：key -> 结果文本
const results = ref({})
// 状态映射：key -> 'running' | 'done' | 'error'
const statusMap = ref({})

// 切换类型只读缓存，不调 API
const currentResult = computed(() => results.value[analysisType.value] || '')
```

**2. 一次性并行生成所有分析**
```javascript
const onAnalyze = async () => {
  onClear()                              // 清空旧结果
  ANALYSIS_TYPES.forEach(t => statusMap.value[t.key] = 'running')

  // 6 个 API 并行触发，每个独立完成
  const tasks = ANALYSIS_TYPES.map(async ({ key, fn }) => {
    try {
      const { data } = await fn(paperId.value)
      results.value[key] = extract(data)  // 存到缓存
      statusMap.value[key] = 'done'
    } catch (e) {
      results.value[key] = `❌ 分析失败：${e.message}`
      statusMap.value[key] = 'error'
    }
  })
  await Promise.allSettled(tasks)
}
```

**3. 移除自动触发**
- `onSearchPaper` 移除 `await onAnalyze()` → 只加载论文信息
- `el-radio-group` 移除 `@change` → 切换只改 v-model，不调 API
- `onMounted` 从 URL 加载 ID 后只查询论文，不自动分析

**4. 状态可视化**
- 每个类型按钮显示状态标记：✓(完成) / ⟳(进行中) / ✗(失败)
- 分析中显示进度："3 / 6"
- 完成后提示："已完成全部 6 项分析，可点击上方按钮切换查看"
- 提供"清空结果"按钮，可重新分析

### 性能对比

| 操作 | 旧逻辑 | 新逻辑 |
|------|--------|--------|
| 切换类型 | 调 LLM API（3-10 秒） | 读缓存（<1ms） |
| 查看 6 种分析 | 6 × 调 API = 30-60 秒 | 1 次并行 = 5-10 秒 |
| 从详情页跳转 | 被迫等 1 次分析 | 立即显示论文信息 |

### 关键教训
1. **昂贵操作的结果必须缓存** — LLM API 调用是耗时操作，结果应该缓存到响应式对象，切换视图只读缓存不重新调用
2. **不要替用户做决定** — 从详情页跳转过来应该让用户决定何时开始分析，而不是自动触发；用户可能只是想看看论文信息
3. **并行优于串行** — 6 个独立的分析任务用 `Promise.allSettled` 并行执行，比串行快 6 倍；每个独立处理失败，互不影响
4. **状态可视化提升体验** — 按钮上的 ✓/⟳/✗ 标记和进度计数让用户清楚知道哪些已完成、哪些在跑

**日期：** 2026-06-28
**严重程度：** 高（3 个独立 bug 叠加）

1. 论文详情页"商业化预测"一直为空
2. 排行榜出现重复论文

### 根因分析

**问题 1：商业化预测为空**
- [PaperDetail.vue](file:///d:/Trae/chat/Paper_Analysis_Engine/frontend/src/views/PaperDetail.vue) 用 `paper.commercial_prediction` + `paper.commercial_score`

**问题 2：排行榜重复论文**
- `ratings` 表的 `paper_id` 列**没有 UNIQUE 约束**（[db.py:45-57](file:///d:/Trae/chat/Paper_Analysis_Engine/backend/db.py#L45-L57)）
- `insert_rating` 用 `INSERT OR REPLACE`，但 `paper_id` 不是 UNIQUE → REPLACE 不生效 → 每次重新评级都新增一行
- `get_all_ratings` 的 SQL `JOIN ratings r ON p.id = r.paper_id` → 一篇论文有 N 条评级就显示 N 次
- 实际数据：611 条评级对应 571 篇论文（40 条重复）

**问题 3：导入不查重**
- [insert_paper](file:///d:/Trae/chat/Paper_Analysis_Engine/backend/db.py#L195) 只按 `id` 去重（`INSERT OR REPLACE`）
- 但不同数据源的同篇论文 ID 不同：OpenAlex 用 `W...`，arXiv 用 `arxiv_...`，领研网用 `linkresearcher_xxx`
- 同一篇论文从 OpenAlex 和 arXiv 导入 → 不同 ID + 相同标题 → 重复入库

### 解决方案

**问题 1 修复** — [api/main.py:get_paper_detail](file:///d:/Trae/chat/Paper_Analysis_Engine/backend/api/main.py#L164-L187)：
- 调用 `predict_commercialization(paper)` 实时计算
- 字段映射：`commercial_score = trl / 9.0`（0-1 范围）
- 文本拼接：`"技术成熟度 TRL X/9 级 | 预计商业化时间：... | 置信度：... | 加速因素：..."`

**问题 2 修复** — [db.py](file:///d:/Trae/chat/Paper_Analysis_Engine/backend/db.py)：
1. `get_all_ratings` SQL 加 `GROUP BY p.id` + `MAX(r.xxx)` 聚合（即使有多条评级也只显示一行）
2. `insert_rating` 改为"先 DELETE 再 INSERT"（避免重复累积）
3. 清理现有重复数据：`DELETE FROM ratings WHERE id NOT IN (SELECT MAX(id) FROM ratings GROUP BY paper_id)` → 611 → 571

**问题 3 修复** — [db.py:insert_paper](file:///d:/Trae/chat/Paper_Analysis_Engine/backend/db.py#L195-L248)：
- 插入前先按标题查重（`LOWER(TRIM(title))` 归一化比较）
- 标题已存在且 ID 不同 → 跳过，返回 False
- 避免不同数据源（OpenAlex/arXiv/领研网）重复导入同篇论文

### 验证
- 排行榜：10 条全是唯一 ID ✅
- 商业化预测：`commercial_score=0.78`（TRL 7/9）+ 完整文本 ✅
- ratings 表：571 条（和 papers 一一对应）✅

### 关键教训
1. **INSERT OR REPLACE 需要 UNIQUE 约束才生效** — SQLite 的 `INSERT OR REPLACE` 只在有 UNIQUE/PRIMARY KEY 冲突时替换；`paper_id` 没有 UNIQUE 约束 → 每次都新增行
2. **JOIN 多对一关系必须 GROUP BY** — `papers JOIN ratings` 是多对一（一篇论文多条评级），必须 `GROUP BY p.id` + 聚合函数
3. **不同数据源的 ID 不一致** — OpenAlex/arXiv/领研网对同一篇论文用不同 ID，必须按标题查重而非仅按 ID
4. **计算字段不入库，用时实时算** — 商业化预测依赖 TRL 估算，每次查详情时实时计算即可，不需要存数据库

AI 只在 `api/main.py` 加了 `load_dotenv()`，但 `llm_analyzer.py` 是独立模块，被其他脚本直接 import 时（如 `cli.py`、`import_papers.py`），`api/main.py` 的 `load_dotenv()` 不会执行。

**坑 2：模块级常量在 import 时立即执行**
```python
# llm_analyzer.py 第 34 行
DEEPSEEK_API_KEY = os.environ.get("DEEPSEEK_API_KEY")
```
这行代码在模块被 import 时立即执行。如果 import 时 `load_dotenv()` 还没调用，`DEEPSEEK_API_KEY` 就是 `None`。

**坑 3：node_modules 漏加 .gitignore**
前端 `node_modules` 目录有 10,000+ 文件（Vue、Element Plus、ECharts 等依赖），不加 `.gitignore` 会污染 git 仓库，clone 时耗时数分钟。

### 解决方案

**修复 1：在 llm_analyzer.py 顶部加 load_dotenv()**
```python
# llm_analyzer.py 第 20-26 行
try:
    from dotenv import load_dotenv
    env_path = Path(__file__).resolve().parent.parent / ".env"
    load_dotenv(env_path)
except ImportError:
    pass
```

关键：
- `Path(__file__).resolve().parent.parent` = backend/../ = **项目根目录**
- `.env` 文件在项目根目录，不在 backend/ 目录

**修复 2：.gitignore 加 node_modules**
```gitignore

frontend/dist/
```

## #27 — P0 功能：论文评分卡 + Zotero 插件 + 中文字体坑

**日期：** 2026-06-30
**严重程度：** 中（功能不完整，但不影响现有功能）

### 问题描述

实施两个 P0 功能：
1. 论文评分卡（matplotlib 生成五维雷达图 PNG）
2. Zotero 插件（保存论文时自动调用 PAE API 标注彩色徽章）

实施过程中遇到中文字体缺失坑 + Edit 工具误删装饰器坑。

### 错误现象

**坑 1：中文字体缺失**
matplotlib 默认用 DejaVu Sans 字体，不含 CJK 字符，导致评分卡上的中文全部显示为方框：
```
UserWarning: Glyph 23398 (\N{CJK UNIFIED IDEOGRAPH-5B66}) missing from font(s) DejaVu Sans.
```

**坑 2：Edit 工具误删装饰器**
用 Edit 工具在 `/api/papers/{paper_id}` 端点后插入新代码时，`old_string` 包含 `@app.post("/api/papers/import")`，但 `new_string` 没有保留这行，导致导入论文 API 没有装饰器：
```python
# 修复前（broken）
def import_paper(paper: PaperImport):  # 缺少 @app.post 装饰器
    ...

# 修复后
@app.post("/api/papers/import")
def import_paper(paper: PaperImport):
    ...
```

### 根因分析

**坑 1 根因**：
- matplotlib 默认字体是 DejaVu Sans，不含 CJK
- 规范里只提到 "Docker 里需 fonts-wqy-zenhei"，但 Windows 本地开发也需要兼容
- 没有跨平台字体检测逻辑

**坑 2 根因**：
- Edit 工具的 `old_string` 必须在文件中唯一，所以包含了下一个端点的装饰器
- 但 `new_string` 只想插入新代码，没把下一个装饰器带回来
- 这类插入操作应该用"前后边界 + 中间插入"模式，而不是"替换上下文"

### 解决方案

**修复 1：跨平台中文字体配置**
```python
# scorecard_generator.py
import matplotlib.font_manager as fm

def _setup_chinese_font():
    candidates = [
        'Microsoft YaHei',     # Windows
        'WenQuanYi Zen Hei',   # Linux (Docker)
        'PingFang SC',         # macOS
        'SimHei',              # Windows 备用
    ]
    available = set(f.name for f in fm.fontManager.ttflist)
    for font_name in candidates:
        if font_name in available:
            plt.rcParams['font.sans-serif'] = [font_name, 'DejaVu Sans']
            plt.rcParams['axes.unicode_minus'] = False
            return font_name
    return None

_ACTIVE_FONT = _setup_chinese_font()
```

```python

    ...
```
