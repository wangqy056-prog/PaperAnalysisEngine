## ADR-001：渐进式数据库架构

**日期：** 2026-06-27

### 背景
项目需要存储论文、评级、标签、引用关系等数据，初期需要选择数据库方案。

### 选项
1. **Neo4j + PostgreSQL** — 图数据库 + 关系数据库
2. **PostgreSQL** — 纯关系数据库
3. **SQLite** — 单文件嵌入式数据库

### 决策
采用**渐进式架构**：SQLite → PostgreSQL → Neo4j

| 阶段 | 数据库 | 触发条件 |
|------|--------|---------|
| MVP | SQLite | 默认，零配置 |
| 增长期 | PostgreSQL | 论文 > 1万篇 |
| 成熟期 | Neo4j | 需要深度组合分析 |

### 理由
- MVP 阶段追求快速验证，SQLite 单文件零配置最简
- Neo4j + PostgreSQL 在第一阶段过度设计，增加部署复杂度
- 数据量达到瓶颈时再迁移，避免过早优化

## ADR-002：数据源优先级

**日期：** 2026-06-28

### 背景
需要选择论文数据源，平衡覆盖率、免费额度、稳定性。

### 选项
1. **Semantic Scholar** — 学术专用，有API Key时5000次/小时
2. **OpenAlex** — 2.5亿+论文，完全免费无限速
3. **arXiv** — 预印本，间隔3秒
4. **Google Scholar** — 无官方API，禁止爬取

### 决策
**主力：OpenAlex + arXiv**
**备用：Semantic Scholar（待配置API Key）**

### 理由
- OpenAlex 完全免费、无限速、覆盖最广（2.5亿+）
- Semantic Scholar 无API Key时限速100次/5分钟，IP易被持续限流
- API Key 申请被拒（QQ/163/Gmail邮箱不被接受）
- arXiv 补充预印本 coverage

### 后续
如获取 Semantic Scholar API Key，可重新启用作为补充数据源。

## ADR-003：Web UI 技术选型

**日期：** 2026-06-28

## ADR-004：LLM 三模型自动降级

**日期：** 2026-06-28

### 背景
论文智能分析需要LLM，需选择模型并保证可用性。

### 选项
| 模型 | 价格 | 速度 | 质量 |
|------|------|------|------|
| DeepSeek V4 Flash | 付费 | 快 | 高 |
| Groq Llama 3.3 70B | 免费1000次/天 | 极快(394 TPS) | 中高 |
| 智谱 GLM-4-Flash | 完全免费 | 中 | 中 |

### 决策
**三模型自动降级机制**

```
降级链：DeepSeek → Groq → GLM（可配置起始模型）

fallback_chain = {
    "deepseek": [_call_deepseek, _call_groq, _call_glm],
    "groq":     [_call_groq, _call_deepseek, _call_glm],
    "glm":      [_call_glm, _call_deepseek, _call_groq],
}
```

### 理由
- DeepSeek 质量最高但付费，作为主力
- Groq 免费且极速，作为第一备用
- GLM 完全免费无限制，作为兜底
- 任一模型失败自动切换，保证服务可用性

### Groq 配置
- Base URL: `https://api.groq.com/openai/v1`（OpenAI兼容）
- Model: `llama-3.3-70b-versatile`
- 免费额度：1000次/天，无需信用卡

## ADR-005：知识图谱可视化方案

**日期：** 2026-06-28

### 背景
需要可视化论文间关系，构建知识图谱。

### 选项
1. **引用网络** — 真实学术引用关系
2. **领域共现** — 基于研究领域
3. **作者合作** — 基于共同作者
4. **关键词共现** — 基于标题关键词
5. **论文相似** — 基于内容相似度

### 决策
**全部实现**，支持5种网络图切换

### 理由
- 引用网络最权威，但需补充抓取数据
- 其他4种基于现有数据立即可用
- 不同视角互补，全面展示论文关系

### 技术方案
- **networkx** 构建图结构（力导向布局）
- **plotly** 交互式可视化（Scatter + 连线）
- 节点大小=度数，颜色=连接数
- 可调节参数：节点数、共现阈值、相似度阈值

### 相似度算法
- Jaccard 相似度（领域+标题+摘要关键词+作者的交集/并集）
- 阈值 0.2 + kNN 兜底（每篇连接最相似的5篇）

## ADR-008：推荐引擎多策略混合

**日期：** 2026-06-28

### 背景
需要为用户提供个性化论文推荐，但单一策略覆盖面不足。

### 选项
1. **纯内容匹配** - 基于用户浏览/收藏的论文领域+关键词
2. **协同过滤** - 需要用户行为数据积累
3. **多策略混合** - 综合多种推荐方式

### 决策
**多策略混合推荐**

支持5种策略：
- content（内容匹配）：基于用户画像的领域+关键词
- rating（高评分）：Top评分论文
- trending（热门）：log(引用) + 评分
- recent（最新）：时间维度
- **hybrid（混合）**：综合以上策略，各来源加权得分

### 权衡
- 优点：覆盖面广，新用户可用（rating/trending不依赖历史）
- 缺点：协同过滤未实现，个性化程度有限
- 后续可加入协同过滤和深度学习推荐

## ADR-009：推送方式多渠道支持

**日期：** 2026-06-28

### 背景
需要支持多种推送渠道，对接不同用户习惯。

### 选项
1. **本地文件** - 最简单，生成Markdown报告
2. **Webhook** - 通用HTTP回调
3. **微信** - 通过PushPlus/Server酱中转
4. **邮件** - SMTP发送

### 决策
**全部支持**

- file: 最简实现，无配置门槛
- webhook: 自动适配企业微信/钉钉/飞书的payload格式

### 权衡
- 优点：覆盖主流推送场景
- 缺点：微信依赖第三方服务，稳定性受限
- 定时服务可常驻运行（`python scheduled_push.py --serve`）或通过 cron/计划任务触发

## ADR-007：Windows 编码兼容方案

**日期：** 2026-06-28

### 背景
Windows 控制台默认 GBK 编码，无法处理论文标题中的特殊字符。

### 决策
所有脚本入口统一添加 UTF-8 重配置：

```python
import sys
if sys.platform == "win32":
    try:
        sys.stdout.reconfigure(encoding='utf-8', errors='replace')
        sys.stderr.reconfigure(encoding='utf-8', errors='replace')
    except Exception:
        pass
```

### 理由
- `errors='replace'` 用替代字符替代无法编码的字符，避免崩溃
- 只在 Windows 平台执行，不影响其他系统
- 已固化为项目模板，所有新脚本默认包含

## ADR-010：Web UI 技术栈迁移（Streamlit → 替代方案）

**日期：** 2026-06-28

### 背景
Streamlit 在 MVP 阶段快速验证了功能，但遇到无法解决的固有限制：
1. 页面切换 DOM 残留（3 次修复仍无法解决）
2. session_state 管理复杂，状态容易串扰
3. JS 注入受限，iframe 干扰渲染
4. 不适合复杂多页面交互应用

### 选项

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **FastAPI + React** | 完全控制 UI、生态丰富、性能好 | 需要前端开发、学习曲线陡 | 大型应用、团队开发 |
| **FastAPI + Vue** | 渐进式框架、中文生态好、上手快 | 需要前端构建工具 | 中大型应用 |
| **NiceGUI** | Python 全栈、基于 Vue/Quasar、无 DOM 残留问题 | 生态较小、复杂场景可能受限 | 中小型应用、Python 开发者 |
| **Dash (Plotly)** | 数据可视化强、Python 全栈 | UI 灵活性不如前端框架 | 数据分析仪表盘 |
| **Flask + HTMX** | 轻量、无需前端构建、服务端渲染 | 交互性较弱 | 简单应用 |
| **Gradio** | 类似 Streamlit、AI 应用友好 | 同样有单文件限制 | AI 演示应用 |

### 决策
**FastAPI + Vue 3**（用户确认采用，2026-06-28）

技术栈组合：
- 后端：FastAPI + uvicorn（45 个 RESTful 端点，自动 Swagger 文档）
- 前端：Vue 3 + Vite + Element Plus + ECharts + vis-network + marked
- 路由：Vue Router 4（`router.afterEach` 实现切换自动滚顶）
- 通信：axios + Vite 代理 `/api` → `http://localhost:8000`

### 理由
- **效果最好**：完全控制 UI 和交互，原生路由彻底解决 Streamlit 的 DOM 残留问题
- 后端代码（db.py/paper_fetcher.py/rating_engine.py/llm_analyzer.py 等）可完全复用
- 只需替换 app.py（Streamlit UI）为 FastAPI 端点 + Vue 前端
- CLI 工具（cli.py）不受影响，继续可用
- 前后端分离，可独立部署，长期可维护性最佳

### 实施结果
- ✅ FastAPI 后端：45 个 API 端点，服务 http://localhost:8000，文档 http://localhost:8000/docs
- ✅ Vue 3 前端：12 个页面，服务 http://localhost:5173，`npm run build` 通过
- ✅ 页面切换无 DOM 残留（Vue Router 原生路由）
- ✅ 自动滚动到顶部（`router.afterEach` + `window.scrollTo(0, 0)`）
- ✅ 修复 5 个 API 导入错误（knowledge_graph / recommendation_engine / search_in_collection / report_generator / scheduled_push）
