## 五、免费用户积累策略 🔥

### 5.1 核心原则

> **先给价值，再收钱。免费版要好用到让用户主动推荐。**

### 5.2 六大获客渠道

#### 渠道 1：学术社区渗透（最有效）

| 平台 | 策略 | 预期效果 |
|------|------|---------|
| **知乎** | 回答"如何高效追踪论文""AI 论文推荐工具"类问题，附工具链接 | 每篇 500-5000 阅读 |
| **即刻/少数派** | 发帖分享"我用 AI 搭了一套论文评级系统" | 精准技术人群 |
| **小红书** | "研究生必装！这个工具让文献综述少花80%时间" | 学生群体裂变 |
| **GitHub** | 开源前端代码 + Star 引流 | 技术口碑 |
| **B站** | 教程视频"从0搭建论文推荐引擎" | 技术爱好者 |

#### 渠道 2：高校合作（最快上量）

```
第一招：联系 3-5 个高校实验室，让 PI 试用
       → 收集反馈 → 迭代 → 扩展到同院系
第二招：发邮件给 arXiv 上近期论文的通讯作者
       → "你的论文在我们系统里评级 X 级，点击看分析"
       → 学术虚荣心驱动转化（类似 ResearchGate 的套路）
```

#### 渠道 3：内容营销（长尾）

- 每周发一篇"本周 AI/量子/生物 论文 Top 10"公众号文章
- 每篇底部附"查看更多排名→"工具链接
- 3 个月积累 50+ 篇文章，搜索引擎自然流量

#### 渠道 4：工具型裂变

```
免费功能：
  ✅ 输入一篇论文 DOI → 自动生成五维雷达图（可分享图片）
  ✅ 输入一个研究方向 → 自动生成研究空白热力图
  ✅ "你的论文评分卡" → 一键分享到朋友圈/群
```

#### 渠道 5：SEO 长尾词

| 关键词 | 月搜索量 | 难度 |
|--------|---------|------|
| 论文推荐工具 | 1000+ | 中 |
| 文献综述工具 | 3000+ | 中 |
| AI 论文分析 | 5000+ | 高 |
| arXiv 热门论文 | 8000+ | 高 |
| 学术论文评级 | 200+ | **低（蓝海）** |

#### 渠道 6：API 开放

开放基础搜索 API（免费额度 1000 次/天），让其他开发者集成，形成生态。

### 5.3 免费→付费转化漏斗

| 属性 | 信息 |

| 团队 | 清华大学唐杰教授团队 |

| 覆盖范围 | 220 个国家和地区，1000+ 高校 |

| 收费模式 | 免费试用 + 会员制 |
| 网址 | https://www.aminer.cn |

**AMiner 的弱点（我们可利用的空白）：**

| AMiner 不做的 | 我们的机会 |
|--------------|-----------|
| 只有引用指标，无多维度量化评级 | 五维评级（学术+商业+创新+复现+组合） |
| 定位学术搜索，不碰商业评估 | TRL + 专利 + 市场三位一体的商业化预测 |
| 重"学者画像"轻"论文价值" | 以论文本身为核心评估对象 |
| 封闭平台，无 API/插件 | 开放 API + Zotero 插件生态 |
| 无定时推送/订阅 | 4 方式 × 4 策略推送体系 |
| 无组合分析/技术路线图 | 协同度计算 + 研究空白识别 |
| 知识图谱聚焦学者关系 | 论文-专利-产品 四层知识图谱 |

**结论：AMiner 是同一赛道的大象，但不是直接竞争对手。它是"学术界的 LinkedIn"——告诉你谁在研究什么。我们要做的是"学术界的 Moody's"——给论文评级、给技术定价。两者定位互补。**

### ⚠️ AMiner 的真实威胁评估

```
如果 AMiner（智谱 GLM 加持）决定做"论文评级"：

他们的优势：
  ✅ 3.3 亿论文数据（我们 571 篇，差距 58 万倍）
  ✅ 1.3 亿学者画像（同名消歧根本不是问题）
  ✅ GLM 大模型（国内最强之一）
  ✅ 清华学术背书 + 1000+ 高校合作
  ✅ 20 年品牌积累

他们为什么还没做？
  🟡 核心定位是"学者画像"——评级需要完全不同的产品逻辑
  🟡 商业评级需要专利数据、TRL 评估、市场分析——这些 AMiner 没有
  🟡 智谱 AI 的精力在通用大模型竞争（vs DeepSeek/通义），不是垂直工具
  🟡 学术搜索赛道不赚钱（AMiner 主要靠政府项目经费，不是 SaaS）

我们怎么活？
  1. 不在"论文数据量"上竞争 → AMiner 的 3.3 亿我们永远追不上
  2. 在"论文-专利-产品"链路深度上竞争 → AMiner 没有这个数据
  3. 在"嵌入式工作流"上竞争 → Zotero 插件 AMiner 没有
  4. 在"商业变现"上竞争 → 我们的评级直接回答"这项技术值多少钱"
  5. 唯快不破 → AMiner 是大组织决策慢，我们可以在 6 周内做出他们没有的功能

最坏情况：
  AMiner 上线"论文评级"功能 → 我们仍然有专利映射 + Zotero 插件 + 
  评分卡分享 + 用户行为数据这四个 AMiner 不会马上有的护城河。
```

## 十三、专利-论文映射 —— 技术方案与成本评估

### 13.1 技术难度矩阵

| 子任务 | 技术栈 | 难度 | 说明 |
|--------|--------|------|------|
| BigQuery 数据导出 | SQL + Python | ⭐⭐ | 写 SQL 查询 → 导出 CSV，1 天内搞定 |
| DOI 正则提取 | Python re | ⭐ | NPL 文本里 DOI 格式相对规范 |
| 标题模糊匹配 | SQLite FTS5 + difflib | ⭐⭐ | 拼写变体、缩写需要容忍度 |
| 作者同名匹配 | Python + 消歧逻辑 | ⭐⭐⭐ | 同名问题是核心难点，见下文 |
| 语义相似度匹配 | embedding + 向量检索 | ⭐⭐⭐ | 计算量大，需要 GPU 或 API |
| 数据管道自动化 | Python + cron | ⭐⭐ | 增量更新逻辑，参考 events 推送 |
| 可视化 | vis-network（已有）| ⭐ | 复用知识图谱组件 |

> **总技术难度：⭐⭐⭐（中等偏高）。核心难点在同名消歧和语义匹配，其他都是标准数据处理。**

### 13.2 同名消歧 —— 最棘手的部分

**问题规模**：

| 场景 | 重名概率 | 我们的论文量 | 预计冲突 |
|------|---------|------------|---------|
| Geoffrey Hinton | ~0% | 1 | 0 |
| "张伟" | ~95% | 可能有 50+ 同名 | 严重 |
| "Li Wei" | ~80% | 可能几十个 | 严重 |
| "Wang Lei" | ~85% | 可能几十个 | 严重 |

**消歧策略（分层加权）**：

```
Layer 1（硬匹配，置信度 99%）：
  ORCID ID 匹配 → 论文 author.orcid == 专利用发明人 ORCID → 直接确定

Layer 2（强信号，置信度 90%）：
  邮箱域名匹配 → author@tsinghua.edu.cn → 发明人机构 == "Tsinghua University" → 高分

Layer 3（传统加权，置信度 70-85%）：
  总分 = 姓名匹配(40) + 机构匹配(25) + 领域匹配(20) + 时间窗口(10) + ORCID/邮箱加分(5)
  
  阈值：≥ 75 → 同一人
         60-74 → 需人工审核
         < 60 → 不是同一人
```

> ⚠️ **诚实预期**：中文重名（张伟/李伟/王磊）即使加上机构匹配，精度也很难超过 70%。这部分映射应标记为"参考信号"而非"确定性关系"，前端展示时加置信度标注。

代码示例：

```python
def match_author_inventor(paper_author, patent_inventor, 
                          paper_affiliation, patent_assignee):
    score = 0
    
    # 1. 姓名匹配（必须一致）
    if normalize_name(paper_author) != normalize_name(patent_inventor):
        return 0
    score += 50
    
    # 2. 机构匹配
    if paper_affiliation and patent_assignee:
        sim = string_similarity(paper_affiliation, patent_assignee)
        if sim > 0.8: score += 30
        elif sim > 0.5: score += 15
    
    # 3. 领域匹配（论文 field vs 专利 IPC 分类）
    if field_overlap(paper_field, patent_ipc_class):
        score += 20
    
    # 4. 时间窗口（专利申请在论文发表后 0-5 年）
    if 0 <= (patent_date - paper_date).days <= 1825:
        score += 10
    
    return score
```
