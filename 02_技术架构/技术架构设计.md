## 二、核心组件

### 2.1 数据接入层

| API | 数据源 | 特点 | 优先级 |
|-----|--------|------|--------|
| **Semantic Scholar API** | 1.7亿+论文 | AI驱动，有影响力评分 | 高 |
| **OpenAlex** | 2.5亿+论文 | 完全开放，数据最全 | 高 |
| **arXiv API** | 预印本 | 实时更新，计算机/物理首选 | 高 |
| **PubMed API** | 生物医学 | 医学领域权威 | 中 |
| **Crossref API** | DOI元数据 | 补充元数据 | 中 |
| **Google Scholar** | 综合搜索 | 覆盖最广，但无官方API | 低 |

### 2.2 数据存储层（渐进式演进）

> **核心原则**：从轻量到重量，按需升级，避免过度设计

#### 第一阶段：SQLite + JSON（零配置）

```
为什么用 SQLite：
• 单文件部署，无需安装
• 零配置启动，适合MVP快速验证
• 支持JSON字段存储复杂结构
• 足够支撑1万篇论文的查询性能

数据结构：
papers.db (SQLite 单文件)
├── papers 表
│   ├── id (TEXT PRIMARY KEY)
│   ├── doi (TEXT)
│   ├── title (TEXT)
│   ├── abstract (TEXT)
│   ├── authors (JSON)        -- ["作者1", "作者2"]
│   ├── metadata (JSON)       -- {"journal": "xxx", "year": 2026}
│   ├── citations (INTEGER)
│   ├── rating (JSON)         -- {"academic": 85, "commercial": 70}
│   ├── tags (JSON)           -- [{"name": "LLM", "category": "tech"}]
│   └── references (JSON)     -- ["doi1", "doi2"]  -- 用SQL递归查引用链
│
├── ratings 表
│   ├── paper_id (TEXT)
│   ├── academic_impact (REAL)
│   ├── commercial_potential (REAL)
│   ├── overall_score (REAL)
│   └── created_at (TEXT)
│
└── tags 表
    ├── id (TEXT)
    ├── name (TEXT)
    ├── category (TEXT)
    └── description (TEXT)

引用关系查询（用 SQL 递归）：
-- 查询某论文的所有上游引用
WITH RECURSIVE citation_chain AS (
    SELECT id, references FROM papers WHERE id = '目标论文ID'
    UNION ALL
    SELECT p.id, p.references FROM papers p, citation_chain c
    WHERE json_array_contains(c.references, p.doi)
)
SELECT * FROM citation_chain;
```

#### 第二阶段：PostgreSQL（扩展阶段）

```
什么时候升级：
• 论文数量 > 1万篇
• 需要并发写入（多人协作）
• 需要复杂查询优化
• 需要全文检索

升级策略：
• 导出 SQLite 数据 → 导入 PostgreSQL
• 代码层只需改连接字符串，业务逻辑不变
```

#### 第三阶段：Neo4j（图谱阶段）

```
什么时候引入：
• 论文节点 > 1万篇
• 组合分析需要复杂图谱遍历
• 需要可视化引用网络
• 需要研究社区图谱

节点类型：
- Paper (论文)
- Author (作者)
- Tag (标签)

关系类型：
- CITES (引用)
- SYNERGY_WITH (协同关系)
```

#### Chroma/Qdrant - 向量数据库

```
用途：
- 论文摘要语义搜索
- 相似论文推荐
- RAG检索增强

什么时候引入：第二阶段（AI分析时）
```

## 三、技术栈

### 后端（渐进式）

| 阶段 | 数据库 | 特点 | 适用场景 |
|------|--------|------|---------|
| **第一阶段** | SQLite + JSON | 零配置，单文件 | MVP验证，命令行工具 |
| **第二阶段** | PostgreSQL | 并发支持，全文检索 | 论文>1万，Web服务上线 |
| **第三阶段** | PostgreSQL + Neo4j | 图谱查询，组合分析 | 组合分析功能开发时 |

- **语言**：Python 3.12
- **框架**：FastAPI（第二阶段引入）
- **向量数据库**：ChromaDB（第二阶段引入）
- **任务队列**：Celery + Redis（第二阶段引入）

### 前端
- **框架**：React + Vite
- **图表**：ECharts / D3.js
- **UI组件**：Ant Design / shadcn/ui
- **图谱可视化**：Neo4j Bloom / React-Force-Graph

### AI/ML
- **LLM**：GLM-5.2 / GPT-4 / Claude 3.5
- **Embedding**：text-embedding-3-small / bge-large-zh
- **RAG框架**：LangChain

### 部署
- **容器**：Docker
- **编排**：Kubernetes (可选)
- **云平台**：阿里云/腾讯云

## 四、数据流（分阶段）

### 第一阶段数据流（MVP）

```
1. 命令行工具启动
2. API调用 → 数据清洗 → 存入 SQLite 单文件
3. 命令行查询 → SQL查询 → 输出结果
4. 评级计算 → 纯Python算法 → 存入 ratings 表
```

### 第二阶段数据流（Web服务）

```
1. API采集 → PostgreSQL 存储
2. 用户搜索 → 前端 → FastAPI → PostgreSQL → 前端展示
3. AI分析 → LLM调用 → 生成标签 → PostgreSQL存储
4. 向量检索 → ChromaDB → 相似论文推荐
```

### 第三阶段数据流（图谱分析）

```
1. 组合分析 → Neo4j图谱遍历 → 协同度计算
2. 引用网络可视化 → Neo4j → 前端图谱组件
3. 研究社区识别 → Neo4j社群检测 → 输出结果
```
